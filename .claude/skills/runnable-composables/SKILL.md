# Runnable Composables Skill

`app/composables/`를 `action`, `sideeffect`, `store`로 분리하는 기준을 적용한다.

## 목적

composable의 책임을 나눠 코드 이해 비용을 낮춘다.
외부 통신과 상태 변경을 테스트 가능하고 추적 가능한 형태로 유지한다.
브라우저 전용 지도 엔진과 화면 상태를 섞지 않고 분리한다.

## 패키지 역할

### `app/composables/action/`

독립적인 유틸리티 기능을 구현한다.

- 외부 API, 브라우저 IO, 전역 상태에 직접 의존하지 않는다
- 입력과 출력이 명확한 순수 로직 또는 그에 가까운 조합 로직을 둔다
- 여러 화면이나 store에서 재사용 가능한 작은 단위 기능을 우선 배치한다
- 예: 트리 탐색, 평탄화, 데이터 변환, 좌표 계산

### `app/composables/sideeffect/`

외부 API 통신 등 부수 효과를 이해하기 쉽고 테스트 가능하게 관리한다.

- 네트워크 요청, 브라우저 API 접근, 타이머, 로깅 등 외부 세계와 상호작용하는 코드를 둔다
- 부수 효과의 실행 조건과 실패 처리 흐름이 드러나야 한다
- 테스트 가능성을 위해 래퍼, 어댑터, 의존성 주입을 우선 고려한다
- 상태 저장은 직접 소유하기보다 `store`와 연결하거나 호출 결과를 반환한다
- 예: Cesium 스크립트 로딩, MapPrime viewer 초기화, DOM 조작, 외부 API 호출

### `app/composables/store/`

데이터의 상태를 관리한다.

- 공유 상태, 로딩 상태, 에러 상태, 파생 상태를 한곳에서 이해할 수 있게 둔다
- 상태 변경 함수는 명시적으로 노출하고 전이 규칙을 예측 가능하게 유지한다
- 외부 통신 자체는 `sideeffect`에 두고, 반영만 `store`에서 처리한다
- 예: `useState` 기반 theme-map 상태, 선택된 노드 상태

## 분리 규칙

- `action`은 계산과 조합에 집중하고, 외부 환경 접근을 포함하지 않는다
- `sideeffect`는 외부 환경 접근을 담당하지만 상태의 최종 소유자가 되지 않는다
- `store`는 상태를 소유하지만 네트워크 세부 구현을 품지 않는다
- 하나의 composable이 세 역할을 동시에 수행하면 먼저 책임을 쪼개는 방향을 검토한다
- 페이지 컴포넌트는 가능하면 composable 조합만 수행하고 `window`, `document`, 외부 API 세부 구현을 품지 않는다
- 브라우저 전용 지도 엔진 접근은 `sideeffect`로 수렴시킨다

## 적용 순서

1. 변경하려는 코드가 유틸, 부수 효과, 상태 중 무엇을 담당하는지 먼저 판별한다.
2. 외부 통신과 상태 변경이 섞여 있으면 `sideeffect`와 `store`로 분리한다.
3. 중복 계산이나 공통 변환 로직이 있으면 `action`으로 추출한다.
4. 페이지나 컴포넌트에서는 필요한 composable을 조합만 하도록 유지한다.
5. 브라우저 전역 타입이 필요하면 별도 타입 선언 파일 또는 경계 파일에서 관리한다.

## 점검 항목

- 외부 API 호출이 `action` 또는 페이지 컴포넌트에 남아 있지 않은가
- 상태 초기화와 갱신 규칙이 `store`에서 한눈에 보이는가
- `sideeffect`가 테스트 가능한 경계로 감싸져 있는가
- 문서와 실제 디렉터리 구조가 같은 기준을 설명하는가
- 지도 엔진 초기화와 DOM 접근이 페이지에 직접 남아 있지 않은가
